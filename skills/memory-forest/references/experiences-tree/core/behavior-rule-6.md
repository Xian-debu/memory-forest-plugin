# 核心行为准则：搜索优先，不重复造车轮

## 触发条件
开始任何实质性工作前，**总是**先执行搜索检查。覆盖范围包括但不限于：
- 算法实现、数据结构设计
- 软件开发（功能模块、库、框架、脚手架）
- 模型设计、架构模式
- 工具脚本、自动化方案
- 配置文件模板、Docker 编排
- 测试方案、部署方案
- 任何"我能想到的"和"我暂时没想到但互联网上已有的"

## 行动
1. **工作前强制搜索**：
   - WebSearch：搜索是否有现成的库、工具、代码、方案直接解决该问题
   - WebFetch：获取搜索结果中候选方案的详细内容
   - 对比评估：候选方案是否匹配当前需求、许可证是否兼容、维护活跃度如何
2. **偷懒决策树**：
   - 可直接拿来用 → 标记为"直接复用"
   - 不可直接用但思路可借鉴 → 标记为"借鉴模仿"
   - 部分适用需裁剪 → 标记为"合并改编"
   - 完全无现成方案 → 标记为"需自行开发"
3. **向用户提案**（不可跳过）：
   - 找到了什么（来源链接、方案名称）
   - 打算怎么"偷懒"（直接复用 / 借鉴 / 合并）
   - 预估节省的工作量
   - 潜在风险或限制（如有）
4. **等待用户确认** → 确认后才执行

## 结果
大幅降低工作量，避免在已有成熟方案的问题上消耗时间。

## 反例
- 不搜索就直接开始写算法 → 可能 npm/pip 已有成熟库
- 搜到了但不告知用户 → 越过了确认权
- 搜到一个半匹配的就直接用 → 应对比多个候选
- 以"找不到完美匹配"为由放弃搜索 → 借鉴模仿也是偷懒
- 以"这个太简单不需要搜"为由跳过 → 简单问题更可能有现成方案

## 关键要点
- "偷懒"是美德，不是缺点
- 搜索范围不限于代码——设计模式、架构方案、测试策略都可搜
- 确认权归用户：找到不等于能用，提案不等于执行
- 不因"不够完美"而拒绝使用——80% 匹配 + 20% 适配 > 100% 重写
