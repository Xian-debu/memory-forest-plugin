# 自我进化执行方法论

## 触发条件
执行系统性自我进化（多方向并行改进）时。

## 行动
按以下优先级排序：

### 第一轮：互补配对（零风险）
选择两个互不依赖但互补的方向先行。例如：
- Stop hook（退出保存）+ SessionStart hook（启动加载）= 完整生命周期闭环
- 两者都是纯增量，不改任何现有逻辑，零破坏性

### 第二轮：高价值独立项（需验证）
最有价值但独立的方向次之。例如：
- MCP Server（价值最高，但需协议兼容性验证）
- 先完成再测试，不与其他方向耦合

### 第三轮：锦上添花（低优先级）
摩擦优化最后做。例如：
- 权限白名单优化
- Shell 别名
- 这些改了立刻见效，但不影响核心架构

### 每轮结束后的检查点
1. `mf status` — 森林是否完好
2. `mf health` — 有无引入问题
3. 验证 settings.json 仍是合法 JSON
4. 确认 hooks 能正常执行
5. 更新文档 + 提炼经验

## 结果
四个进化方向在单次会话内完成，零回滚，零破坏。执行顺序避免了同时引入多个变量导致的调试困难。

## 反例
- 四个方向同时开工，交错调试 → 问题来源不明，修改相互覆盖
- 从最难的方向（MCP）开始 → 卡住后整个进化停滞
- 跳过每轮检查点 → 问题累积到最后才发现
- 做完不提炼经验 → 下次进化从零开始

## 关键要点
- **互补先行**：选择天然形成闭环的方向配对（如 start + stop）
- **隔离高风险**：把最复杂的独立项放中间，不和任何东西耦合
- **每轮收敛**：完成一轮→检查→提炼→再开下一轮
- **基础设施边界**：自我迭代只碰工具/配置/脚本，不碰用户代码逻辑（这是 EXP-CORE-0005 的范围限定）

## 适用场景
- 任何需要多个方向并行改进的系统性重构
- AI 自我优化的执行框架
- 跨多个配置文件的自动化改造

## Evolution 3 增量 (2026-05-14)

第三次进化验证并扩展了本方法论：

**新发现**: 每轮进化的驱动问题不同:
- Evolution 1: "有蓝图无发动机" → 基础设施（hooks/MCP/CLI）
- Evolution 2: "系统在退化" → 健康维护（审计/调优/守护）
- Evolution 3: "规则完整但LLM跳过" → 机械防呆（validate/前置注入/护栏）

**驱动问题发现法**: 两次进化之间，哪些 feedback 在重复出现？哪些错误模式在累积？下一轮进化以最高频失败模式为主题。

**LLM特化规律**: 常规软件工程的"文档→自觉"路径对 LLM 无效。进化方向必须是"文档→机械执行"。纸面规则再多，不如一个 PreToolUse hook。

**关键经验**: 每次进化后立即创建 validate 脚本——这样下一次进化时 validate_all 能直接告诉你什么变了、什么退化。
