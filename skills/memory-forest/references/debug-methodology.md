# 调试方法论

## 流程框架

### 阶段 1：复现
```
目标：可稳定复现 Bug
方法：
  - 记录精确的触发步骤
  - 构建最小可复现用例（MRE）
  - 确认环境依赖（特定版本？特定数据？特定时序？）
如果不能复现 → 不猜测修复，继续收集信息
```

### 阶段 2：定位
```
目标：缩小到具体代码行/条件
方法（按顺序尝试）：
  1. git bisect（若已知在哪个版本引入）
  2. 二分排除：注释掉一半代码，确认 Bug 还在哪边
  3. 插桩打印：在关键路径打印状态
  4. 交互式调试：python -m pdb / node inspect
  5. 静态分析：以 tech-methodology.md 的视角审查嫌疑代码
```

### 阶段 3：根因分析
```
目标：理解为什么出错（不只看表象）
方法：
  - 5 Whys：逐层追问"为什么"，直到触及根因
  - 对比分析：什么情况触发？什么情况不触发？差异是什么？
  - 时间线重建：事件在时间上的顺序是否导致问题？
```

### 阶段 4：修复
```
目标：修复根因而非症状
方法：
  - 最小修复：只改必要的代码
  - 防回归：添加自测用例确保同样 Bug 不再出现
  - 影响分析：修复是否影响其他依赖此处行为的代码？
```

### 阶段 5：验证
```
目标：证明 Bug 已修复且未引入新问题
方法：
  - 使用阶段1的 MRE 验证修复
  - 运行完整自测套件
  - 运行相关模块的集成测试
```

## 常见 Bug 模式速查

| 症状 | 常见原因 | 优先检查 |
|------|---------|---------|
| `NoneType has no attribute` | 未检查可选值 | 调用链中哪个函数返回了 None |
| `IndexError / out of range` | 空集合或索引计算错误 | 集合长度、索引来源 |
| `KeyError` | 字典键不存在 | 键的来源、默认值策略 |
| 死循环 | 循环条件永远为真 | 循环变量的更新逻辑 |
| 数据竞态 | 并发访问共享状态 | 锁范围、原子性、happens-before |
| 内存泄漏 | 引用未释放 | 循环引用、事件未解绑、闭包捕获 |
| 连接超时 | 网络不可达或慢 | DNS、防火墙、目标端口 |
| 返回数据不对但无报错 | 逻辑错误 | 数据流追踪、不变量检查 |

## 调试技术工具箱

### 最小可复现用例构建
```python
# 从完整流程中提取最小触发代码
# 原则：每步去掉一个元素，若 Bug 还在则删除该元素
# 最终剩下 < 20 行的触发代码
```

### 二分排除
```python
# 在可疑的代码段中
# 先注释掉后半段，运行看 Bug 是否还在
# 若还在 → 前半段有问题
# 若消失 → 后半段有问题
# 递归直到定位到具体行
```

### 插桩打印
```python
# 在关键路径用 print 或 logging 打印状态
# 格式: [FUNCTION_NAME] variable = value
# 打印到 stderr 避免干扰 stdout 输出
import sys
print(f"[func_name] var={var!r}", file=sys.stderr)
```

### 时间线重建
```
对于时序敏感的 Bug（并发、异步、事件驱动）：
1. 在关键事件点记录时间戳
2. 重建事件发生顺序
3. 检查是否发生意外的事件交错
```

## 不可修复的情况
若经完整流程仍无法定位：
- 明确告知用户：已尝试的方法、排除的范围、剩余可能性
- 请求用户提供更多上下文或复现步骤
- 不编造修复方案
