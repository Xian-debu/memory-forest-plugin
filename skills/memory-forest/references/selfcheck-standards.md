# 自检规范与集成规范

## 自检流程（每次代码修改后执行）

### 第一层：语法与类型检查
```
Python:  python3 -m py_compile <file>  （至少编译检查）
Node:    node --check <file>
Go:      go build ./...
```
- 必须零错误通过
- 若项目有 linter 配置，也需通过

### 第二层：模块自测
```
Python:  python3 <module>.py  （触发 __main__ 自测块）
Node:    node <module>.js
Go:      go test ./...
```
- 所有自测必须 PASS
- 若模块无自测块，至少导入测试：`python3 -c "import module_name"`

### 第三层：逻辑审查
- 重新阅读修改的 diff，逐行检查：
  - 命名是否准确描述意图
  - 边界条件是否处理
  - 安全红线是否触碰
  - 是否引入了不必要的依赖或抽象

### 第四层：报告
- 向用户报告：
  1. 修改了什么文件（路径 + 行号）
  2. 为什么这样改
  3. 自检结果（通过/失败/未覆盖）
  4. 若有不能完全验证的部分，明确声明

## 集成规范

### 集成前检查
- 每个模块独立通过自检
- 模块间接口文档清晰（至少注释标注参数类型和返回值）
- 数据流可追踪（从入口到出口的路径完整）

### 集成测试要求
- 至少包含一个端到端测试（覆盖完整的数据流路径）
- 测试失败场景：错误输入、超时、依赖模块不可用
- Docker 化项目：在容器中运行集成测试

### 集成顺序
```
1. 数据库/存储层  →  2. 模型层  →  3. 业务逻辑层  →  4. API 层  →  5. 前端
```
- 每层验证完成后才进入下一层
- 不跳跃集成

### 多服务集成
- 使用 `docker-compose` 编排
- 每个服务有健康检查端点（`/health` 或 `/ping`）
- 启动顺序通过 `depends_on` 或 `wait-for-it.sh` 控制
- 集成测试至少覆盖：服务发现、正常请求、断连恢复

### 集成后清理
- 停止所有测试容器
- 清理测试数据
- 不保留临时文件在仓库中

## 回归检查
- 修改前先确认相关模块的现有测试通过
- 修改后重跑相关模块测试
- 若现有测试失败，优先修复而非跳过
