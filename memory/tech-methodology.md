# 技术方法论

## 视角体系

### 1. 逻辑追踪（Logic Tracing）
**目的**：追踪代码的实际执行路径，建立心智模型
**方法**：
- 从入口函数开始，逐层展开调用链
- 记录每层的：输入假设 → 处理逻辑 → 输出承诺
- 标记分支条件和合并点
- 检查：每个分支是否可达？是否有死代码？
**触发**：理解陌生代码、Bug 定位、代码审查

### 2. 抽象验证（Abstraction Verification）
**目的**：验证抽象层次是否合理，是否存在泄露
**方法**：
- 识别每层的抽象边界
- 检查：下层概念是否穿透到上层？（抽象泄露）
- 检查：是否存在中间人抽象？（仅转发无变换）
- 检查：抽象命名是否准确反映其职责？
- 规则：N 层抽象中，每层必须带来明确收益
**触发**：架构设计、重构、代码审查

### 3. 数学推理（Mathematical Reasoning）
**目的**：验证算法正确性、循环不变式、状态转换
**方法**：
- 循环：确认循环不变式在每次迭代前后保持
- 递归：确认递归终点可达且基例正确
- 状态机：枚举所有状态转移，检查是否封闭
- 数值：检查溢出、精度损失、除零
- 概率：检查随机算法期望/尾界是否符合要求
**触发**：算法实现、复杂条件逻辑、并发代码

### 4. 冗余探查（Redundancy Detection）
**目的**：发现重复逻辑、死代码、过度抽象
**方法**：
- AST 级相似度检测（同构但不相同的重复）
- 信息熵分析：同一信息在多处编码？
- 调用频率分析：参数组合中有多少实际被使用？
- 检查：两个函数是否总是一起被调用？（考虑合并）
- 检查：配置值是否只被读取一次？（考虑内联）
**触发**：代码审查、优化、重构

### 5. 纯机器视角（Machine-Eye View）
**目的**：从 CPU/内存/缓存/指令流水线的角度审视代码
**方法**：
- CPU 缓存行对齐（hot path 数据结构）
- 分支预测友好性（likely/unlikely 分支）
- 内存布局（struct padding, cache line sharing）
- 指令级并行度（数据依赖链长度）
- 系统调用成本（context switch 频率）
**触发**：性能关键代码、系统编程、C/C++/Rust

### 6. 数据流视角（Data Flow Perspective）
**目的**：追踪数据从入口到出口的完整生命周期
**方法**：
- 绘制 DFD（Data Flow Diagram）：Source → Process → Sink
- 标记每个节点的数据变换和格式
- 检查：是否有隐式数据依赖（全局变量、环境变量）？
- 检查：数据是否在传递中被不必要地拷贝？
- 检查：敏感数据路径是否被隔离（密码、token）？
**触发**：新功能设计、安全审计、集成调试

### 7. 控制流分析（Control Flow Analysis）
**目的**：理解所有可能的执行路径
**方法**：
- 绘制 CFG（Control Flow Graph）
- 识别循环（loop header, back edge, exit）
- 检查：异常路径是否被正确处理？
- 检查：资源在异常路径中是否被释放？
- 圈复杂度评估：> 10 则考虑拆分
**触发**：复杂函数审查、并发代码

### 8. 依赖图分析（Dependency Graph Analysis）
**目的**：理解模块间耦合关系
**方法**：
- 构建模块依赖矩阵
- 识别循环依赖（绝对禁止）
- 识别高扇入模块（修改风险高）
- 识别高扇出模块（稳定性要求高）
- 计算依赖深度（越长越脆弱）
**触发**：架构评估、模块拆分、重构规划

### 9. 状态机验证（State Machine Verification）
**目的**：验证有状态组件的状态转换完整性
**方法**：
- 列出所有合法状态
- 列出所有合法转换（状态 × 事件 → 新状态）
- 检查：是否所有状态对每个事件都有定义？
- 检查：是否存在不可达状态？是否存在死锁状态？
- 检查：是否有竞态导致的状态不一致？
**触发**：状态管理代码、协议实现、连接管理

### 10. 不变量检查（Invariant Checking）
**目的**：识别和验证代码中隐含的不变量
**方法**：
- 显式化不变量：数据结构约束、关系约束
- 检查：每个修改点是否保持不变量？
- 检查：并发修改是否打破不变量？
- 添加 assert 强制执行关键不变量
**触发**：数据结构设计、并发数据结构

### 11. 故障模式分析（Failure Mode Analysis）
**目的**：系统性地思考"什么可能出错"
**方法**：
- 对每个外部依赖问：它失败了会怎样？
- 对每个网络调用问：超时了会怎样？返回异常值会怎样？
- 对每个文件操作问：权限不够会怎样？磁盘满了会怎样？
- 对每个资源申请问：用完后是否保证释放？
- 对每个并发操作问：乱序到达会怎样？
**触发**：可靠性要求高的代码、网络编程、文件操作

### 12. 复杂度分析（Complexity Analysis）
**目的**：评估算法的时间和空间效率
**方法**：
- 时间复杂度：最好/平均/最坏，确定瓶颈操作
- 空间复杂度：内存使用峰值、是否原地
- 通信复杂度：网络调用次数、数据量
- 实用性判断：输入规模 × 每操作成本 是否可接受
**触发**：算法选择、性能优化、大数据处理

## 视角组合使用指南

| 任务类型 | 优先使用的视角 |
|----------|---------------|
| 代码审查 | 逻辑追踪 + 冗余探查 + 不变量检查 |
| Bug 定位 | 逻辑追踪 + 数据流视角 + 故障模式分析 |
| 性能优化 | 纯机器视角 + 复杂度分析 + 数据流视角 |
| 安全审计 | 数据流视角 + 控制流分析 + 故障模式分析 |
| 架构设计 | 抽象验证 + 依赖图分析 + 状态机验证 |
| 算法实现 | 数学推理 + 复杂度分析 + 不变量检查 |
| 并发编程 | 状态机验证 + 不变量检查 + 故障模式分析 |
| 重构 | 冗余探查 + 抽象验证 + 依赖图分析 |
| 集成调试 | 数据流视角 + 控制流分析 + 故障模式分析 |

## 执行模式

### 静默应用（默认）
在编写代码时自动在"后台"运行相关视角，不显式输出分析过程，仅将结果反映在代码质量中

### 显式调用
用户明确要求时（如"审查这段代码"），输出结构化的分析结果
格式：
```
## [视角名称]
- 发现: [具体问题]
- 位置: file:line
- 建议: [修复方案]
```

### 自检触发
每次代码修改完成后，自动以相关视角重新审视修改，作为自检流程的一部分
