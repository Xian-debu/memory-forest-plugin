# 设计规范

## 核心原则
1. **简单优先**：能用内置库解决的问题不引入第三方库；能用单文件解决的不建包
2. **组合优于继承**：偏好函数组合和接口实现，避免深层继承链
3. **显式优于隐式**：数据流可见，不依赖隐式全局状态或魔法方法
4. **接口隔离**：模块间通过最小接口通信，不暴露内部实现细节
5. **失败早报**：在入口处校验，快速失败，不传播错误状态

## Plan-then-Code 流程

### 触发条件（满足任一即进入计划模式）
- 新功能实现（> 20 行新代码）
- 跨文件修改（> 2 个文件）
- 架构决策（选型、模式、数据流设计）
- 存在 2 种以上可行方案
- 用户未给出精确实现细节

### 流程
```
1. EnterPlanMode  →  探索代码库、理解现状
2. 设计 1-2 个方案  →  对比利弊、推荐最优
3. ExitPlanMode   →  用户审批
4. 实现代码       →  按方案执行
5. 自检验证       →  证明修改正确
```

### 不必进入计划模式的情况
- 单行/少量 bug 修复
- 已有明确指令的简单实现
- 纯研究/探索型问题

## 架构模式

### Python 项目
- **单体脚本**（< 100 行）：单文件，无包结构
- **小型项目**（100-500 行）：按功能分包 `core/`, `utils/`, `models/`
- **中型项目**（500-2000 行）：分层架构 `api/`, `service/`, `repository/`, `models/`
- **大型项目**（> 2000 行）：DDD 或微服务拆分

### 微服务/分布式
- 每个服务独立目录、独立 Dockerfile
- 服务间通过 HTTP REST 或 MQTT/消息队列通信
- 每个服务有独立的测试套件

### 数据流设计
- 明确数据从何处进入系统（入口）
- 明确数据经过哪些转换（管道）
- 明确数据最终写入哪里（出口）
- 画 ASCII 流程图辅助理解（复杂场景）

## 设计检查清单（每次设计时过一遍）
- [ ] 是否有更简单的方案能达到同样效果？
- [ ] 新增的依赖是否必要？
- [ ] 错误路径是否被处理？
- [ ] 是否存在竞态条件或并发问题？
- [ ] 数据是否在正确的层级持有？
- [ ] 接口是否足够小且稳定？
