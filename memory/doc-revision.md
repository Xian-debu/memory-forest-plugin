# 文档修订规范

## 修订触发条件

### 自动触发
- 代码实践中发现规范不适配当前场景
- 规范之间存在逻辑冲突
- 用户纠正了我的行为模式
- 环境变更（新工具、新语言版本、新权限模型）
- 发现了新的通用技术视角或方法论

### 手动触发
- 用户明确要求修改某规范
- 用户批准了规范修改提议

## 修订流程

### 1. 提议阶段
```
发现需求 → 评估影响范围 → 向用户提议（一句话说明：改什么、为什么）
```
- 不静默修改规范
- 给出修改前后的对比（diff 形式或文字摘要）

### 2. 编写阶段
- 读取目标文件（确认当前状态）
- 用 Edit 工具精确修改，不重写整文件
- 保持 frontmatter 同步更新
- 若新增文件，检查未与现有文件职责重叠

### 3. 一致性检查
- 扫查 MEMORY.md 是否需要更新索引
- 扫查 doc-relationship.md 是否需要更新关系
- 扫查被引用方：其他规范中对本条规范的引用是否仍然准确
- 若新增规范文件，必须更新 MEMORY.md 索引

### 4. 提交阶段
- 向用户报告修订完成
- 说明：修改了哪些文件、对现有规范的影响

## 修订类型

| 类型 | 操作 | 示例 |
|------|------|------|
| **追加** | Edit 在末尾添加节 | 新增一种技术视角 |
| **细化** | Edit 替换段落 | 补充具体例子 |
| **修正** | Edit 替换错误规则 | 纠正过时的工具名 |
| **废弃** | Edit 标记为 `[DEPRECATED]` + 指向新规范 | 规范被更完善的取代 |
| **合并** | 两文件内容合并到一个，另一个删除 | 两个规范高度重叠 |
| **拆分** | 一个大文件拆为多个，更新索引 | 单个文件超过 200 行 |

## 冲突解决

### 规范间冲突
当两个规范对同一情况给出矛盾指引：
```
1. 按优先级裁决: 安全 > 功能正确 > 用户指令 > 设计 > 风格 > 文档格式
2. 应用高优先级规范
3. 标记低优先级规范为待修订
4. 提议修改低优先级规范以消除冲突
```

### 规范与用户指令冲突
```
用户明确指令优先于规范（因为规范服务于用户目标）
但需：
1. 明确告知用户指令与规范 X 冲突
2. 说明冲突点
3. 让用户确认是否仍按指令执行
4. 执行后记录，供后续规范修订参考
```

### 规范与代码现实冲突
```
若代码库现状不符合规范（如既有项目使用不同风格）：
1. 遵循项目既有风格（一致性优先）
2. 不在本次修改中大规模对齐风格
3. 若有价值，提议专项"风格对齐"任务
```

## 新文件创建检查清单
- [ ] 未与现有文件职责重叠
- [ ] frontmatter 完整（name, description, type）
- [ ] type 字段正确（user/feedback/project/reference）
- [ ] MEMORY.md 添加索引条目
- [ ] doc-relationship.md 更新关系
- [ ] 文件名符合 doc-standards.md 命名约定

## 文件删除检查清单
- [ ] 确认内容已完全迁移到其他文件（若合并）
- [ ] 确认无其他规范引用该文件
- [ ] 从 MEMORY.md 移除索引
- [ ] 从 doc-relationship.md 移除关系

## 版本意识
- 规范不设版本号，以文件修改时间为版本依据
- 重大修订后，在文件末尾添加 `<!-- last-major-revision: YYYY-MM-DD, 变更摘要 -->`
- 历史记录通过 git 追踪（若环境为 git 仓库）
