# 基准功能验证规范

## 原则
- **无验证，不宣称完成**：不能仅凭代码 diff 声称任务完成
- **证据驱动**：必须提供运行输出、测试结果或截图作为完成证明
- **最低可观测**：至少运行一次修改涉及的代码路径

## 基准验证矩阵

| 修改类型 | 最低验证要求 |
|----------|-------------|
| 新 Python 文件 | `python3 file.py` 运行自测块，零错误 |
| 修改 Python 函数 | 触发该函数的代码路径至少执行一次 |
| 新 API 端点 | `curl` 或 `python3 -c "import requests; ..."` 实际调用 |
| 修改配置/环境变量 | 读取并打印配置值确认生效 |
| Bug 修复 | 复现 bug 的代码 → 修复前失败 → 修复后通过 |
| Docker 相关 | `docker build` 成功 + `docker run` 验证 |
| Bash 脚本 | 实际执行一次，检查退出码 |
| HTML/CSS 变更 | 若无法浏览器测试，至少静态验证语法正确 |
| 多文件重构 | 每个修改文件至少通过编译/导入检查 |

## 验证优先使用的方式
1. **模块自测块**（`__main__` 中的测试）
2. **单文件直接运行**（`python3 file.py`）
3. **交互式验证**（`python3 -c "..."` 单行验证）
4. **curl/HTTPie**（API 验证）
5. **Docker 运行**（容器化项目）

## 验证失败处理
- 若自测失败：修复 → 重新验证 → 通过后才报告
- 若无法验证（需要浏览器、外部服务等）：明确告知用户未验证部分
- 若验证环境受限：给出用户侧验证步骤

## 报告模板
```
## 修改摘要
- 文件: path/to/file.py:42 (修改原因)
- 文件: path/to/other.py:10-15 (修改原因)

## 自检结果
- [x] 语法检查通过
- [x] 模块自测通过 (3/3 tests)
- [x] 集成测试通过
- [ ] 前端验证: 未执行（需要浏览器，请用户验证）

## 运行证据
(paste 实际输出)
```
