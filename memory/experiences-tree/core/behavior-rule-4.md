# 核心行为准则：经验迭代合并与垃圾识别

## 触发条件
每次提取新经验时，以及定期（按 MF-SPEC 垃圾回收周期）审查经验池时。

## 行动
1. **新经验入库前**：搜索已有经验，检查是否存在可合并项
2. **合并条件**（满足任一）：
   - 两条经验的核心行动指向同一模式
   - 触发条件高度重叠
   - 一条经验是另一条的特化/泛化
3. **合并操作**：保留更凝练的版本，将另一条的来源引用合并进来，删除冗余文件
4. **垃圾识别**（满足任一）：
   - 引用次数 = 0 且 heartbeat > 90天
   - 经验被后续经验覆盖（旧经验是子集）
   - 经验关联的源节点全部 ARCHIVED 且经验本身从未被引用
   - 经验描述的场景不再可能发生（环境已变更）
5. **垃圾处理**：提炼残余有用信息 → 合并到存活经验 → 删除垃圾节点
6. **定期审查**：每 90 天遍历全部经验，执行上述流程

## 结果
经验池保持精简、准确、不过时。

## 反例
- 经验只增不减，最终变成垃圾场
- 两条本质相同的经验分别存储，检索时不知该用哪条
- 环境变更后旧经验仍保留，导致错误指导

## 关键要点
- "不变"是经验的敌人，必须持续迭代
- 合并时优先保留更具体、更可操作的经验
- 删除前检查所有引用，确保无断裂引用
