# 核心行为准则：多步骤任务必须先建立记忆森林子节点

## 触发条件
任何任务满足以下**任一**条件时，必须在开始执行前建立记忆森林子节点：
- 包含 3 个或以上独立子任务
- 预计跨多个会话才能完成
- 子任务之间有依赖关系（A 完成后才能做 B）
- 用户明确要求"逐个"、"依次"、"一个一个"执行

## 行动
1. **建节点先于写代码**：
   - 在对应项目树下为每个子任务创建 LEAF 节点
   - 每个 LEAF 节点包含：目标、源文件、目标文件、验证标准
   - 项目 ROOT 的 `children` 列表必须包含所有新节点 ID
2. **状态同步**：
   - 开始子任务时：LEAF status → ACTIVE，同时更新 Claude Task 为 in_progress
   - 完成子任务时：LEAF status → COMPLETED，heartbeat 刷新
   - 遇到阻塞时：LEAF 内写明阻塞原因，不删除节点
3. **中断恢复设计**：
   - 每个 LEAF 节点是可独立恢复的最小单元
   - 节点内包含足够的上下文（源路径、目标路径、关键决策），使恢复者无需回溯对话历史
   - `resumeSessionId` 字段标记最后一个处理该节点的会话
4. **完成后归档**：
   - 全部子任务完成后，父 BRANCH/ROOT 更新状态汇总
   - 提炼跨子任务的模式到 experiences-tree

## 为什么
- `.claude/tasks/` 目录是会话级存储，下一会话不自动加载，不能作为中断恢复的唯一依据
- 记忆森林是跨会话持久化的，下一会话的启动流程会扫描 MEMORY.md → 项目 ROOT → 子节点
- 没有子节点的任务列表 = 会话死亡时任务也死亡
- 本次会话的教训：10 个分类器迁移任务在 `7ad82639` 会话中建立了 Claude Task 列表(1-10.json)，但未建记忆森林子节点，导致会话中断后新会话完全不知道任务存在，直到用户手动引导才找到 `/root/.claude/tasks/7ad82639/` 残余

## 反例
- 任务列表只建在 Claude Task 系统里，不在记忆森林建节点 → 会话中断即失忆
- 建了节点但没写关键上下文（源路径、验证方法）→ 恢复者仍然需要猜测
- 子任务粒度太粗（一个节点覆盖 5 个独立步骤）→ 中断时无法确定精确断点
- 完成后不更新父节点 children 列表 → 下一会话扫描不到子节点

## 关键要点
- 记忆森林是"外部大脑"，Claude Task 是"工作内存"——两者必须同步
- 建节点的成本远低于会话中断后重建上下文的成本
- 一个 LEAF = 一个可独立验证的完成单元 = 一个安全的断点
